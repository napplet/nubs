  Edit NUB-IFC.md on the nub-ipc branch to add channel support. The changes are:

  **A. Update API Surface** — Add channel methods to the NappletIfc interface (per D-03 through D-09):

  ```typescript
  interface NappletIfc {
    // Topic-based pub/sub (existing)
    emit(topic: string, payload?: unknown): void;
    on(topic: string, callback: (event: IfcEvent) => void): Subscription;

    // Point-to-point channels (new)
    channel: {
      open(target: string): Promise<ChannelHandle>;     // via ifc.channel.open
      list(): Promise<ChannelInfo[]>;                    // via ifc.channel.list
      broadcast(payload: unknown): void;                 // via ifc.channel.broadcast
    };
  }

  interface ChannelHandle {
    readonly id: string;           // shell-assigned channel ID
    readonly peer: string;         // peer dTag
    emit(payload: unknown): void;  // via ifc.channel.emit
    on(callback: (event: ChannelEvent) => void): Subscription;
    close(): void;                 // via ifc.channel.close
  }

  interface ChannelEvent {
    channelId: string;
    sender: string;    // sender dTag
    payload: unknown;
  }

  interface ChannelInfo {
    id: string;
    peer: string;      // peer dTag
  }
  ```

  Method descriptions:
  - **`channel.open(target)`** — Opens a point-to-point channel to a napplet identified by its dTag (per D-03). The shell validates the target and checks ACL on open. Returns a `ChannelHandle` on success. If the target is not found or ACL-denied, the promise rejects. Per D-05, the shell validates once on open — subsequent messages flow without per-message checking.
  - **`channel.list()`** — Returns active channels for this napplet (per D-08).
  - **`channel.broadcast(payload)`** — Sends a message to all open channels at once (per D-07). Fire-and-forget.
  - **`ChannelHandle.emit(payload)`** — Sends a message to the channel peer (per D-06). Fire-and-forget.
  - **`ChannelHandle.on(callback)`** — Receives messages from the channel peer (per D-06). The callback receives a `ChannelEvent` with channelId, sender dTag, and payload.
  - **`ChannelHandle.close()`** — Tears down the channel (per D-09). Both sides are notified.

  **B. Add Wire Protocol entries** — Add a new "### Channels" subsection within Wire Protocol, after the existing topic messaging entries:

  | Type | Direction | Payload fields |
  |------|-----------|----------------|
  | `ifc.channel.open` | napplet -> shell | `id`, `target` (dTag string) |
  | `ifc.channel.open.result` | shell -> napplet | `id`, `channelId`?, `peer`? (dTag), `error`? |
  | `ifc.channel.emit` | napplet -> shell | `channelId`, `payload`? |
  | `ifc.channel.event` | shell -> napplet | `channelId`, `sender` (dTag), `payload`? |
  | `ifc.channel.broadcast` | napplet -> shell | `payload`? |
  | `ifc.channel.list` | napplet -> shell | `id` |
  | `ifc.channel.list.result` | shell -> napplet | `id`, `channels` (ChannelInfo array) |
  | `ifc.channel.close` | napplet -> shell | `channelId` |
  | `ifc.channel.closed` | shell -> napplet | `channelId`, `reason`? (string) |

  Key design notes for the channel wire protocol:
  - `ifc.channel.open` uses `id` for correlation; result returns `channelId` (shell-assigned).
  - `ifc.channel.emit` has no `id` field — fire-and-forget like `ifc.emit`.
  - `ifc.channel.event` has no `id` field — shell-initiated delivery, carries `channelId` + `sender`.
  - `ifc.channel.broadcast` has no `id` field — fire-and-forget to all open channels.
  - `ifc.channel.close` has no `id` — the `channelId` identifies the target.
  - `ifc.channel.closed` is shell-initiated notification (graceful close, peer destroyed, ACL revoked).


  **D. Add "## Channels vs Topics" section** (replaces old "## Relationship to NUB-PIPES"):

  Title: "## Channels vs Topics"

  Content: Explain the distinction — topics are loose-coupled pub/sub (any napplet subscribes to any topic, sender doesn't know receivers). Channels are pre-authorized point-to-point connections (the shell validates once on `ifc.channel.open`, then messages flow with no per-message checking). Use topics for infrequent coordination, UI commands, and state sync. Use channels for sustained data streams, real-time collaboration, and command channels between specific napplets.

  **E. Update Shell Behavior** — Add channel-specific MUST/MAY clauses:
  - The shell MUST respond to `ifc.channel.open` with `ifc.channel.open.result` carrying the same `id`.
  - The shell MUST assign a `channelId` (opaque identifier) on successful channel open.
  - The shell MUST validate the target dTag exists and has a live iframe before opening a channel.
  - The shell MUST forward `ifc.channel.emit` messages to the channel peer as `ifc.channel.event`.
  - The shell MUST NOT perform per-message validation on channel messages after open (per D-05) — only the initial open is validated.
  - The shell MUST send `ifc.channel.closed` to both sides when a channel is torn down (graceful close, iframe removal, or ACL revocation).
  - The shell MUST respond to `ifc.channel.list` with `ifc.channel.list.result` listing the napplet's active channels.
  - The shell MUST deliver `ifc.channel.broadcast` to all open channel peers, excluding the sender.
  - The shell MAY enforce ACL checks on `ifc.channel.open` (e.g., restrict which dTags a napplet can open channels to).
  - The shell MAY impose a maximum number of concurrent channels per napplet.
  - The shell MUST clean up channel state when a napplet iframe is destroyed, sending `ifc.channel.closed` with reason `"peer destroyed"` to the surviving endpoint.

  **F. Update Security Considerations** — Add channel-specific notes:
  - Channel authorization is validated once on `ifc.channel.open`. After open, messages flow without per-message shell validation. A compromised browser extension could forge messages on an open channel — this is an accepted trust boundary, the same as topic-based IFC.
  - Channel IDs are opaque. Napplets cannot enumerate or guess other channels' IDs. This prevents channel hijacking.
  - `ifc.channel.broadcast` reaches all open channel peers. Napplets SHOULD NOT send sensitive data via broadcast. For confidential communication, use a named channel.

  **G. Remove all NUB-PIPES references** — Delete the "## Relationship to NUB-PIPES" section and any sentences mentioning NUB-PIPES. Replace with the "Channels vs Topics" section from step D.

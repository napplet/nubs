Rewrite NUB-IFC.md from scratch following the validated structure from NUB-STORAGE and NUB-RELAY. The spec defines inter-frame communication (topic-based pub/sub between napplets) routed through the shell.

**Header** (same format as STORAGE/RELAY):
- NUB ID: NUB-IFC
- Namespace: `window.napplet.ifc`
- Discovery: `shell.supports("ifc")`

**Description:**
One paragraph explaining that NUB-IFC provides topic-based publish/subscribe for communication between napplets. Sandboxed iframes cannot communicate directly (no `allow-same-origin`). The shell routes messages between napplets using typed `ifc.*` messages. This is loose coupling -- the sender does not know who receives the message. Mention the shell also uses IFC topics for internal coordination.

**API Surface** (per D-06 -- TypeScript interface with `// via ifc.*` annotations):
```typescript
interface NappletIfc {
  emit(topic: string, payload?: unknown): void;         // via ifc.emit
  on(topic: string, callback: (event: IfcEvent) => void): Subscription;  // via ifc.subscribe
}

interface IfcEvent {
  topic: string;
  sender: string;    // sender dTag (napplet type identifier, per D-03)
  payload: unknown;
}

interface Subscription {
  close(): void;     // via ifc.unsubscribe
}
```

- `emit(topic, payload?)` -- Broadcasts a message to all napplets subscribed to the given topic. Fire-and-forget -- no delivery confirmation. The shell identifies the sender via `MessageEvent.source` (per SPEC.md).
- `on(topic, callback)` -- Subscribes to messages on a topic. The callback receives an `IfcEvent` with the topic, sender dTag, and payload. Returns a `Subscription` handle with `close()` to unsubscribe. Multiple subscriptions to the same topic are independent.

**Wire Protocol** (per D-01 -- `ifc.emit`, `ifc.subscribe`, `ifc.unsubscribe`, `ifc.event`):

Table of message types:

| Type | Direction | Payload fields |
|------|-----------|----------------|
| `ifc.emit` | napplet -> shell | `topic`, `payload`? |
| `ifc.subscribe` | napplet -> shell | `id`, `topic` |
| `ifc.subscribe.result` | shell -> napplet | `id` |
| `ifc.unsubscribe` | napplet -> shell | `id`, `topic` |
| `ifc.event` | shell -> napplet | `topic`, `sender` (dTag), `payload`? |

Key design notes:
- `ifc.emit` has NO `id` field -- it is fire-and-forget, no acknowledgment (Claude's discretion: fire-and-forget semantics).
- `ifc.subscribe` uses `id` for correlation so the shim can confirm the subscription was registered.
- `ifc.subscribe.result` confirms registration (carries the `id` back).
- `ifc.unsubscribe` uses `id` for correlation.
- `ifc.event` has NO `id` field -- it is a shell-initiated delivery containing `topic`, `sender` (dTag string per D-03), and optional `payload`.

Examples (same `->` / `<-` format as RELAY/STORAGE):

Emit:
```
-> { "type": "ifc.emit", "topic": "profile:open", "payload": { "pubkey": "abc123..." } }
```
(No response -- fire-and-forget.)

Subscribe:
```
-> { "type": "ifc.subscribe", "id": "a1", "topic": "profile:open" }
<- { "type": "ifc.subscribe.result", "id": "a1" }
```

Event delivery:
```
<- { "type": "ifc.event", "topic": "profile:open", "sender": "social-feed", "payload": { "pubkey": "abc123..." } }
```

Unsubscribe:
```
-> { "type": "ifc.unsubscribe", "id": "b2", "topic": "profile:open" }
```

Error handling: `ifc.subscribe.result` MAY include an `error` field (string) if the shell rejects the subscription.

**Topic Conventions** (per D-02 -- preserved from existing spec):

| Prefix | Direction | Meaning |
|--------|-----------|---------|
| `shell:*` | napplet -> shell | Commands sent by a napplet to the shell (e.g., `shell:state-get`) |
| `napplet:*` | shell -> napplet | Responses/notifications from shell to napplet (e.g., `napplet:state-response`) |
| `{domain}:*` | bidirectional | Domain-scoped messages between napplets (e.g., `profile:open`, `chat:open-dm`) |

State that these conventions are advisory. The shell routes by topic match, not by prefix parsing.

**Shell Behavior** (MUST/MAY clauses per D-04, D-07):
- The shell MUST route `ifc.emit` messages to all napplets subscribed to the matching topic.
- The shell MUST identify the sender via `MessageEvent.source` and include the sender's `dTag` in delivered `ifc.event` messages. (per D-03, D-07)
- The shell MUST NOT deliver `ifc.event` back to the emitting napplet (sender exclusion, per D-04).
- The shell MUST respond to `ifc.subscribe` with `ifc.subscribe.result` carrying the same `id`.
- The shell MUST honor `ifc.unsubscribe` by removing the subscription for that topic.
- The shell MAY intercept specific topic prefixes (e.g., `shell:*`) for internal command handling rather than routing them to other napplets.
- The shell MAY enforce ACL checks on IFC capabilities.

**Security Considerations** (per D-07):
- Sender identity is shell-enforced via `MessageEvent.source` mapping to napplet identity (per SPEC.md). There is no per-message signing -- the shell's sender identification is the trust boundary.
- Topic namespaces are not enforced -- any napplet can emit on any topic. The shell MAY restrict topics via ACL.
- Payloads are opaque to the shell. Receiving napplets are responsible for validating payload content.
- Sender exclusion prevents echo loops but does not prevent a napplet from emitting messages on any topic. Receivers should check the `sender` dTag if sender identity matters for their use case.
- For high-frequency point-to-point communication where per-message overhead matters, use NUB-PIPES instead.

**Relationship to NUB-PIPES** (brief section, same as existing):
NUB-IFC is for loose-coupled, topic-based pub/sub. NUB-PIPES is for tight-coupled, point-to-point connections. Both coexist: IFC for infrequent coordination (UI commands, state sync), pipes for sustained data streams.

CRITICAL: Do NOT include any references to kind 29003, IFC_PEER, signed events, NIP-01 subscriptions, Schnorr signatures, delegated session keys, or pubkeys in napplet-facing sections. The old spec's crypto model is completely replaced by `MessageEvent.source` identity.
